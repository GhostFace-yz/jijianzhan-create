# v1.0 AI 适配器层 PRD

## 1. 执行摘要

### 1.1 问题陈述

极简栈创造平台需要接入多个 AI 供应商（文本、图像、视频、TTS、音乐）。如果每个业务模块直接调用不同供应商的 API，会导致代码耦合、切换成本高、错误处理不统一，难以支持节点级模型热切换。

### 1.2 解决方案

构建统一的 AI 适配器层，为文本、图像、视频、TTS、音乐五类任务定义标准接口。每个供应商实现对应 Adapter，系统启动时注册到 Adapter Pool。业务模块通过统一的 `generate()` 接口调用，支持按节点绑定模型和热切换。

### 1.3 成功标准

- 新增供应商 Adapter 时，业务代码零改动。
- 节点级模型切换成功率 100%。
- 所有 Adapter 实现统一接口覆盖率 100%。
- Adapter 健康检查失败时自动切换 fallback 成功率 ≥ 95%。

## 2. 用户体验与功能

### 2.1 用户画像

- **平台管理员**：需要灵活配置不同供应商的 API Key 和参数。
- **内容创作者**：需要在每个生成节点选择最适合的模型。

### 2.2 用户故事

1. **作为** 平台管理员，**我希望** 新增一个供应商时无需修改业务代码，**以便** 快速扩展 AI 能力。
2. **作为** 内容创作者，**我希望** 在每个生成节点右上角选择模型，**以便** 根据需求切换。
3. **作为** 内容创作者，**我希望** 某个模型失败时自动切换到备用模型，**以便** 不中断创作流程。
4. **作为** 平台管理员，**我希望** 查看每个 Adapter 的健康状态，**以便** 及时发现服务异常。

### 2.3 验收标准

#### 五类 Adapter 接口

- [ ] `AITextAdapter.generateText(prompt, systemPrompt, schema, config): Promise<{ content: string; usage: TokenUsage }>`。
- [ ] `AIImageAdapter.generateImage({ prompt, negativePrompt, referenceImages, seed, width, height, stylePreset }, config): Promise<{ url: string; seed: number; usage: ImageUsage }>`。
- [ ] `AIVideoAdapter.generateVideo({ imageUrl, referenceImages, duration, cameraMove, motionDescription, audioUrl, faceEnhancement }, config): Promise<{ url: string; duration: number; usage: VideoUsage }>`。
- [ ] `AITTSAdapter.generateSpeech({ text, voiceId, emotion, speed }, config): Promise<{ url: string; duration: number; usage: TTSUsage }>`。
- [ ] `AIMusicAdapter.generateMusic({ styleTags, emotionSequence, duration, instrumentPref }, config): Promise<{ url: string; duration: number; usage: MusicUsage }>`。

#### 适配器注册表

- [ ] 系统启动时加载所有 Adapter，根据 `provider` ID 路由。
- [ ] 新增供应商只需实现接口并注册，无需修改业务代码。
- [ ] 支持通过配置文件启用/禁用供应商。

#### 模型绑定

- [ ] 每个项目有 `ModelBindings`，包含 outline、script、storyboard、character_img、scene_img、video、video_fallback、tts、music 等节点的默认模型配置。
- [ ] 每个生成节点右上角显示当前模型，点击下拉切换其他已配置模型。
- [ ] 切换后提示「下次生成将使用新模型」。

#### 健康检查与降级

- [ ] 每个 Adapter 实现 `healthCheck()` 方法。
- [ ] 健康检查失败时标记 Adapter 不可用。
- [ ] 调用失败时根据错误类型决定是否切换 fallback：可用性错误切换，非可用性错误抛出。

#### 统一错误码与日志

- [ ] 所有 Adapter 返回统一错误码和错误信息。
- [ ] 记录每次调用的 provider、model、耗时、token/用量、状态。

### 2.4 非目标

- 本阶段不实现 Adapter 的自动负载均衡（除 fallback 外）。
- 本阶段不实现跨供应商结果融合。
- 本阶段不实现供应商计费对账系统。

## 3. AI 系统要求

### 3.1 工具要求

- 各类 AI 供应商的 SDK 或 HTTP API。
- 统一的请求/响应封装。
- 错误分类和重试机制。

### 3.2 评估策略

- 单元测试覆盖所有 Adapter 接口方法。
- 集成测试：模拟每个供应商成功/失败响应。
- 新增一个测试 Adapter，验证业务代码零改动。

## 4. 技术规格

### 4.1 架构概述

```
业务模块 → AdapterPool → 按 provider 路由 → 具体 Adapter → 供应商 API
                ↓
         healthCheck / fallback / 用量统计
```

### 4.2 集成点

- **API**: 内部服务接口，不直接暴露给前端。
- **配置**: `config/ai-adapters.yaml` 或数据库表 `ai_providers`。
- **监控**: 调用日志表 `ai_call_logs`。

### 4.3 安全与隐私

- API Key 通过环境变量或密钥管理服务存储。
- 不向业务模块暴露原始 API Key。
- 调用日志中敏感参数脱敏。

## 5. 风险与路线图

### 5.1 分阶段发布

- **MVP**: 五类 Adapter 接口定义、注册表、基础路由、健康检查。
- **v1.1**: Fallback 切换、统一错误码、调用日志。
- **v2.0**: 自动负载均衡、结果融合、用量计费对账。

### 5.2 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 不同供应商参数差异大 | 高 | 统一接口 + 各 Adapter 内部转换 |
| 供应商 API 变更 | 中 | Adapter 层隔离，只需更新对应 Adapter |
| API Key 泄露 | 高 | 密钥管理 + 调用日志脱敏 |

## 6. 依赖关系

- 前置依赖：无。
- 被依赖模块：所有生成模块（大纲、角色、场景、脚本、分镜图、TTS、配乐、视频）。

## 7. 验收测试要点

1. 为每个 Adapter 类型编写单元测试，验证接口签名。
2. 新增一个 mock Adapter，验证注册表自动加载和业务代码零改动。
3. 模拟 Adapter 失败，验证 fallback 切换逻辑。
4. 验证健康检查接口返回状态正确。
5. 验证调用日志包含 provider、model、耗时、状态，且 API Key 已脱敏。
