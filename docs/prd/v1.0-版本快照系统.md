# v1.0 版本快照系统 PRD

## 1. 执行摘要

### 1.2 解决方案

为所有可编辑内容（项目元信息、大纲、角色、场景、脚本、分镜节点、生成结果）提供版本快照能力。每次 AI 生成、用户编辑、重新生成均创建新版本，支持 diff 对比、回滚到任意历史版本，并记录变更来源和使用的 AI 模型。

### 1.3 成功标准

- 所有可编辑实体的版本快照创建成功率 100%。
- 版本回滚成功率 100%。
- Diff 对比准确率 ≥ 95%。
- 版本历史加载时间 ≤ 500ms（单个实体 50 个版本以内）。

## 2. 用户体验与功能

### 2.1 用户画像

- **创作者**：需要撤销不满意修改，比较不同版本差异。
- **审核者**：需要追溯变更来源（AI 生成 / 用户编辑 / 重新生成）。

### 2.2 用户故事

1. **作为** 创作者，**我希望** 每次编辑后自动生成版本快照，**以便** 随时回滚。
2. **作为** 创作者，**我希望** 查看两个版本之间的 diff，**以便** 了解具体变化。
3. **作为** 创作者，**我希望** 回滚到任意历史版本，**以便** 撤销错误修改。
4. **作为** 审核者，**我希望** 看到每次变更的来源和使用的 AI 模型，**以便** 追溯问题。

### 2.3 验收标准

#### 版本快照数据结构

- [ ] `version_id`：v1, v2, v3...（每个实体独立编号）。
- [ ] `source`：`ai_generated` / `user_edited` / `ai_regenerated`。
- [ ] `content`：该版本的完整内容。
- [ ] `parent_version`：父版本 ID（可选）。
- [ ] `diff`：与父版本的差异记录，`Record<string, 'changed' | 'added' | 'removed'>`。
- [ ] `edited_by`：用户 ID（用户编辑时）。
- [ ] `ai_model`：AI 模型配置（AI 生成时）。
- [ ] `prompt_override`：用户给 AI 的额外指令（可选）。
- [ ] `created_at`：创建时间。

#### 触发条件

- [ ] 用户手动编辑字段后自动保存新版本。
- [ ] AI 生成完成后自动保存新版本。
- [ ] 用户点击「重新生成」后保存新版本。
- [ ] 用户点击「确认」锁定后保存一个稳定版本。

#### 版本历史 UI

- [ ] 每个可编辑节点提供「时钟图标」入口。
- [ ] 点击后展开版本时间轴，显示版本号、来源、时间、操作人/模型。
- [ ] 支持选择两个版本进行对比，高亮差异字段。
- [ ] 提供「回滚到此版本」按钮，回滚前二次确认。

#### Diff 计算

- [ ] 自动计算当前版本与父版本的字段级差异。
- [ ] 支持对象、数组、字符串的差异标记。
- [ ] 对于大文本字段，支持行级 diff。

#### 回滚机制

- [ ] 回滚时创建新版本，内容为历史版本内容，`source` 标记为 `user_edited`。
- [ ] 回滚后触发变更传播，下游节点标记为「需检查」。
- [ ] 回滚操作记录审计日志。

### 2.4 非目标

- 本阶段不实现跨实体的全局版本快照（如一键回滚整个项目到某时刻）。
- 本阶段不实现版本分支合并。
- 本阶段不实现版本快照的自动清理策略。

## 3. AI 系统要求

本模块为基础设施，不直接调用 AI。

## 4. 技术规格

### 4.1 架构概述

```
用户编辑/AI 生成 → 捕获当前内容 → 计算 diff → 存储 version_snapshot → 关联到实体 → 提供历史查询/回滚
```

### 4.2 集成点

- **API**: `GET /api/v1/projects/{id}/entities/{entity_type}/{entity_id}/versions`、`POST /api/v1/projects/{id}/entities/{entity_type}/{entity_id}/versions/{version_id}/rollback`、`GET /api/v1/projects/{id}/entities/{entity_type}/{entity_id}/versions/compare`。
- **数据库**: `version_snapshots` 表（或按实体分表）。
- **业务模块**: 所有可编辑实体在保存时调用版本快照服务。

### 4.3 安全与隐私

- 版本快照按项目隔离。
- 敏感字段（如用户 ID）在对外展示时按需脱敏。

## 5. 风险与路线图

### 5.1 分阶段发布

- **MVP**: 版本快照创建、历史查看、diff 对比、回滚。
- **v1.1**: 变更来源追踪、AI 模型记录、审计日志。
- **v2.0**: 全局项目快照、分支管理、自动清理策略。

### 5.2 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| 版本数据量过大 | 中 | 仅保存 diff 或按需全量存储，后续增加清理策略 |
| Diff 计算性能差 | 中 | 对大数据分片计算，使用高效 diff 算法 |
| 回滚后下游状态不一致 | 高 | 回滚触发变更传播，标记下游为「需检查」 |

## 6. 依赖关系

- 前置依赖：无。
- 被依赖模块：所有可编辑业务模块（项目、大纲、角色、场景、脚本、分镜节点）。

## 7. 验收测试要点

1. 编辑一个角色字段，验证自动生成新版本。
2. 触发 AI 重新生成，验证版本 `source` 为 `ai_regenerated` 且记录 `ai_model`。
3. 选择两个版本对比，验证 diff 高亮正确。
4. 回滚到历史版本，验证内容恢复且下游节点标记为「需检查」。
5. 验证版本历史加载时间 ≤ 500ms（50 个版本）。
6. 测试并发编辑时的版本冲突处理。
