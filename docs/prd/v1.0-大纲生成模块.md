# v1.0 大纲生成模块 PRD

## 1. 执行摘要

### 1.1 问题陈述

短剧创作的第一步是将一个模糊的创意扩展为结构化的剧情大纲，包括世界观、主要冲突、角色、场景和分集规划。创作者通常缺乏将创意系统化整理的工具，导致后续创作方向不清。

### 1.2 解决方案

用户在项目详情页手动点击「生成大纲」后，系统将项目元信息和创意描述组装为结构化 Prompt，调用文本 AI 返回严格 JSON 格式的大纲数据。生成后提供可视化预览和编辑能力，并在确认前执行剧本医生自洽性检查。

### 1.3 成功标准

- 大纲生成调用成功率 ≥ 95%。
- AI 返回有效 JSON 的比例 ≥ 90%。
- 剧本医生检查覆盖率 100%（角色位置、道具状态、情绪曲线、服装连续性）。
- 用户对大纲的编辑操作 100% 支持版本快照。

## 2. 用户体验与功能

### 2.1 用户画像

- **编剧型创作者**：希望 AI 提供初稿，然后手动精修。
- **导演型创作者**：关注角色和场景的完整性，需要可视化检查。

### 2.2 用户故事

1. **作为** 编剧，**我希望** 手动触发 AI 生成大纲，**以便** 获得结构化的剧情框架。
2. **作为** 编剧，**我希望** 在预览页查看世界观、角色、场景和集数时间轴，**以便** 全面了解 AI 生成结果。
3. **作为** 编剧，**我希望** 直接编辑任意模块的文字内容，**以便** 调整不符合预期的部分。
4. **作为** 编剧，**我希望** 重新生成单个集的内容，**以便** 在不破坏整体大纲的情况下优化局部。
5. **作为** 导演，**我希望** 在确认大纲前看到自洽性检查报告，**以便** 避免后续剧情逻辑错误。

### 2.3 验收标准

#### 生成触发

- [ ] 项目详情页提供「生成大纲」按钮，仅在项目状态为 `draft` 或 `outlining` 时可用。
- [ ] 点击后系统组装 Prompt：系统指令（资深编剧）+ 项目元信息 + 创意描述 + JSON Schema 约束。
- [ ] 要求 AI 严格返回 JSON，禁止 markdown 或额外说明。

#### 输出结构

- [ ] AI 输出必须包含：`world_setting`、`main_conflict`、`characters[]`、`locations[]`、`episode_count`、`episodes[]`。
- [ ] 每个 `episode` 至少包含：`episode_number`、`title`、`summary`、`key_events`、`featured_characters`、`featured_locations`。

#### 预览与编辑

- [ ] 预览页包含四个卡片区域：世界观、角色列表、场景列表、集数时间轴。
- [ ] 集数时间轴为横向卡片，点击单集卡片可展开查看详情并修改。
- [ ] 每个模块右上角提供「编辑」按钮，编辑后实时保存草稿。
- [ ] 支持「重新生成大纲」（全量重生成，需二次确认）。
- [ ] 支持「重新生成选中集」（仅刷新选中集内容）。

#### 确认与锁定

- [ ] 点击「确认大纲，进入角色创建」后大纲锁定。
- [ ] 锁定后仍可修改，但触发警告提示相关集可能需要重新生成。
- [ ] 确认后项目状态更新为 `outlining` 或 `asset_prep`。

#### 变更传播

- [ ] 大纲修改后，已生成的脚本和分镜节点标记为「需检查」，不自动删除。
- [ ] 系统顶部横幅提示受影响范围（如"3 个节点需重新生成分镜图"），提供「一键标记」和「忽略」按钮。

#### 剧本医生（自洽性检查）

- [ ] 大纲确认前强制执行检查。
- [ ] 检查项 1：角色位置矛盾（同一角色同一时间出现在两个地点）。
- [ ] 检查项 2：道具状态追踪一致性。
- [ ] 检查项 3：情绪曲线合理性（不能连续 5 集无起伏）。
- [ ] 检查项 4：服装连续性。
- [ ] 输出检查报告：红色=错误必须修复，黄色=警告建议修复，绿色=通过。
- [ ] 存在红色错误时不允许进入下一阶段。

### 2.4 非目标

- 本阶段不实现自动根据大纲生成角色圣经或场景圣经（需用户确认后触发）。
- 本阶段不实现大纲的 AI 续写或扩展。
- 本阶段不实现多人实时协作编辑大纲。

## 3. AI 系统要求

### 3.1 工具要求

- 文本 AI Adapter（如 Kimi、DeepSeek、GPT4o）。
- JSON Schema 约束输出。
- Prompt 模板管理。

### 3.2 评估策略

- 抽取 20 个测试创意，人工评估大纲结构完整性和叙事合理性。
- 目标：结构完整率 ≥ 90%，叙事合理性评分 ≥ 4/5。
- 自动化检查：JSON 解析成功率、必填字段覆盖率。

## 4. 技术规格

### 4.1 架构概述

```
用户点击生成 → ProjectService 组装上下文 → TextAdapter.generateText() → AI 返回 JSON → 解析存储 → 预览页渲染
```

### 4.2 集成点

- **API**: `POST /api/v1/projects/{id}/outline/generate`、`PUT /api/v1/projects/{id}/outline`、`POST /api/v1/projects/{id}/outline/validate`。
- **数据库**: `projects` 表的 `outline` 字段（JSONB），`version_snapshots` 表。
- **AI 适配器层**: 通过 `TextAdapterPool` 路由到绑定的文本模型。

### 4.3 安全与隐私

- Prompt 中不暴露用户隐私信息。
- AI 输出进行 JSON Schema 校验，防止脏数据入库。
- 用户编辑内容同样进行长度和敏感词校验。

## 5. 风险与路线图

### 5.1 分阶段发布

- **MVP**: 手动生成、JSON 输出、基础预览和编辑、二次确认。
- **v1.1**: 剧本医生检查、单集重生成、变更影响提示。
- **v2.0**: 多版本对比、AI 辅助续写、情绪曲线可视化。

### 5.2 技术风险

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI 不返回合规 JSON | 高 | JSON Schema + 重试 + 解析失败回退到文本提取 |
| 大纲过长超出模型上下文 | 中 | 分阶段生成：先世界观/角色/场景，再逐集生成 |
| 自洽性检查误报 | 中 | 允许用户标记误报并忽略 |

## 6. 依赖关系

- 前置依赖：项目创建模块。
- 被依赖模块：角色圣经、场景圣经、单集脚本生成。

## 7. 验收测试要点

1. 使用 5 个不同题材项目生成大纲，验证 JSON 结构和字段完整性。
2. 测试单集重生成，验证仅目标集内容变化。
3. 测试剧本医生检查：构造角色位置矛盾和情绪平线数据，验证报告正确性。
4. 测试版本快照：每次编辑后生成新版本，支持回滚。
5. 测试变更传播：修改大纲后相关下游节点正确标记为「需检查」。
